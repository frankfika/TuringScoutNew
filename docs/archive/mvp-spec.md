# MVP 早期方案草稿

> 归档说明：本文档是 MVP 阶段的早期方案草稿，已被 02-v1-prd.md 和 03-v1-system-design.md 扩展替代。

## MVP v0.1 定义（原始版本）

### 核心功能（仅 3 项）
1. **AI 项目 Trending 列表**：GitHub Trending 中 AI 相关项目，手动筛选
2. **项目详情页**：项目基本信息 + GitHub 数据同步
3. **简单筛选**：按语言、时间范围

### 技术方案
- 前端：Next.js
- 数据库：PostgreSQL
- 数据来源：GitHub API（唯一数据源）
- 部署：Vercel
- 更新频率：每天 1 次

### 不做
- 无用户系统
- 无提交功能
- 无多源数据
- 无评分算法（仅按 star 排序）
- 无搜索
- 无后台

### 时间估计
- 1 人 × 2 周

## 为什么这个方案被淘汰

1. **数据源单一**：仅靠 GitHub Trending 无法提供足够价值，与其他工具无差异
2. **无评分算法**：纯按 star 排序无法发现"正在上升"的项目
3. **无用户贡献**：数据质量和覆盖面完全依赖自动化，无法覆盖长尾机会
4. **无差异化**：和 GitHub Trending 本身区别太小

## 从 MVP v0.1 到 V1 的演化

| 维度 | MVP v0.1 | V1 |
|------|---------|-----|
| 数据源 | 仅 GitHub | GitHub + HN + HuggingFace + 用户提交 |
| 评分 | 按 star 排序 | 多维加权评分 |
| 内容类型 | 仅项目 | 项目 + 机会 |
| 用户参与 | 无 | Scout 提交 + 积分 |
| 管理 | 无后台 | 完整后台 |
| 时间 | 2 周 | 8-10 周 |
| 团队 | 1 人 | 1-2 人 |

## 经验教训

MVP 不应过于简陋以至于无法验证核心假设。V1 的"100+ 项目 + Scout 系统 + 评分算法"是最小可验证单元。
