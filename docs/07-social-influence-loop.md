# 07 — Scout / 创作者社交影响力循环

> **V1 不实现任何社交功能。** 本文档描述的是 V2+ 的社交影响力循环设计。V1 阶段的数据完全由 AI 自动抓取，没有用户提交、没有 Scout 积分、没有创作者系统。

## 一、核心理念

TuringScout 的社交层不是附加功能，而是产品质量和数据质量的保障机制。通过公开的积分和排行系统，激励用户贡献高质量数据，形成自我强化的信任网络。

## 二、核心循环

```
                    ┌──────────────────────┐
                    │   项目/机会存在       │
                    └──────────┬───────────┘
                               │
            ┌──────────────────┼──────────────────┐
            ▼                  ▼                   ▼
   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
   │ Scout 发现    │  │ Creator 解释  │  │ 平台验证排序  │
   │ (提交链接)    │  │ (内容创作)    │  │ (评分/审核)   │
   └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
          │                 │                  │
          └─────────┬───────┴──────────────────┘
                    ▼
          ┌──────────────────────┐
          │   公开积分 / 排行     │
          └──────────┬───────────┘
                     │
          ┌──────────┴───────────┐
          ▼                      ▼
   ┌──────────────┐    ┌──────────────────────┐
   │ Scout/Creator│    │ 更多人参与贡献        │
   │ 获得地位     │    │ (看到差距/想上榜)     │
   └──────┬───────┘    └──────────┬───────────┘
          │                       │
          └───────────┬───────────┘
                      ▼
               (循环继续)
```

### 循环驱动力

| 节点 | 驱动力 | 为什么有效 |
|------|--------|-----------|
| 项目/机会存在 | AI 行业高速发展 | 每天都有新项目、新机会 |
| Scout 发现 | 积分 + 地位 + 社区认可 | 先发优势（第一个发现加分） |
| Creator 解释 | 积分 + 流量 + 专业形象 | 内容获得平台曝光 |
| 平台验证排序 | 算法 + 人工审核 | 保证质量，建立信任 |
| 公开积分 | 透明 + 可比较 + 可分享 | 社交货币效应 |
| 获得地位 | 排行榜排名 + 公开认可 | 人类对地位的本能追求 |
| 更多人参与 | 差距驱动 + 利益驱动 | 看到别人上榜也想上 |

## 三、三层排行榜联动

### 3.1 排行榜架构

```
        ┌─────────────────────────────┐
        │     机会榜 (Opportunity)     │  ← 核心价值层
        │     "有什么值得关注的"       │
        └─────────────┬───────────────┘
                      │ 关联
        ┌─────────────┴───────────────┐
        │      项目榜 (Project)        │  ← 基础数据层
        │     "哪些项目在上升"          │
        └─────────────┬───────────────┘
                      │ 发现/验证
        ┌─────────────┴───────────────┐
        │   Scout 影响力榜 (Scout)     │  ← 社区信任层
        │     "谁值得信任"             │
        └─────────────────────────────┘
```

### 3.2 三榜联动机制

| 联动关系 | 机制 | 效果 |
|---------|------|------|
| Scout → 机会 | Scout 提交并被采纳 → 机会上架 | 高质量 Scout 贡献更多上榜机会 |
| 机会 → 项目 | 项目下有上榜机会 → 项目热度提升 | 好机会为好项目引流 |
| 项目 → Scout | 项目上榜 → 最早发现的 Scout 获认可 | 先发优势 |
| Scout → 项目 | 高信用 Scout 提交 → 项目初始可信度更高 | 信任传递 |
| 机会 → Scout | 机会被验证有效 → 提交者信用增加 | 结果导向的信用积累 |

## 四、贡献类型分类

### 4.1 贡献类型定义

| 类型 | 描述 | 基础积分 | 价值 |
|------|------|---------|------|
| **发现 (Discovery)** | 提交一个新的项目或机会 | 10 | 拓展数据库 |
| **验证 (Verification)** | 确认或否认某个机会仍有效 | 5-20 | 保证数据新鲜度 |
| **解释 (Explanation)** | 撰写项目解读、教程、分析 | 30-100 | 增加内容深度 |
| **基准测试 (Benchmark)** | 提供项目性能测试数据 | 50-150 | 独特数据价值 |
| **纠错 (Correction)** | 纠正过期、错误信息 | 15-30 | 维护数据质量 |
| **充实 (Enrichment)** | 补充项目元信息（语言、分类等） | 5-10 | 完善数据 |

### 4.2 积分权重因素

同一贡献类型的积分差异取决于：

| 因素 | 权重影响 | 说明 |
|------|---------|------|
| **时效性** | ×1.0 - ×2.0 | 第一个发现者获得最高积分 |
| **项目重要性** | ×0.5 - ×3.0 | 热门项目贡献积分更高 |
| **证据质量** | ×0.5 - ×2.0 | 附 Github 链接 > 仅文字描述 |
| **社区认可** | ×0.0 - ×1.5 | 其他人点赞/确认可加成 |
| **历史信用** | ×0.5 - ×1.5 | 高信用 Scout 贡献自动加权 |

## 五、积分计算规则

### 5.1 发现积分

```
DiscoveryScore = base_points × timeliness × project_importance × evidence_quality

base_points = 10
timeliness：第1个提交 = 2.0，前5 = 1.5，前10 = 1.2，之后 = 1.0
project_importance：热门(评分>80) = 3.0，中等(50-80) = 2.0，一般(<50) = 1.0
evidence_quality：附GitHub链接 = 2.0，附官网 = 1.5，仅文字 = 1.0
```

### 5.2 验证积分

```
VerificationScore = base_points × accuracy_bonus × recency_need

base_points = 5
accuracy_bonus：验证结果被后续确认 = 4.0，未被质疑 = 1.0，被纠正 = -1.0
recency_need：机会即将过期 = 2.0，活跃机会 = 1.0，长期机会 = 0.5
```

### 5.3 创作者积分（V2+）

```
CreatorScore = base_points × content_quality × relevance × engagement

base_points = 30 (解读) / 50 (教程) / 100 (基准测试)
content_quality：人工审核评级 (1-3)
relevance：与在榜项目的相关度 (0.5-2.0)
engagement：外部互动量 (likes, shares, comments 归一化)
```

### 5.4 灌水和作弊惩罚

| 违规类型 | 判定 | 惩罚 |
|---------|------|------|
| 重复提交 | 相同 URL 在短时间内多次提交 | 不积分 + 警告 |
| 低质量轰炸 | 大量提交被拒绝 | 降低提交权重 |
| 虚假信息 | 故意提交错误信息 | 积分清零 + 封禁 |
| 刷分 | 多账号协同刷分 | IP/设备关联检测 + 封禁 |
| 抄袭内容 | Creator 内容被证明抄袭 | 内容下架 + 积分扣回 |

## 六、信用机制

### 6.1 信用分构成

```
CreditScore = 积分总量 (40%) + 采纳率 (25%) + 社区认可度 (15%)
            + 活跃度 (10%) + 信誉年龄 (10%)

采纳率 = 被采纳的提交数 / 总提交数
社区认可度 = 获得其他 Scout 的点赞/确认数
活跃度 = 近 30 天贡献天数 / 30
信誉年龄 = min(注册天数 / 365, 1.0)  # 最高 1 年封顶
```

### 6.2 信用等级

| 等级 | 信用分 | 特权 |
|------|--------|------|
| 🥉 Scout | 0-500 | 基础提交权限 |
| 🥈 Verified Scout | 501-2000 | 提交自动加权 + 审核优先级 |
| 🥇 Top Scout | 2001-5000 | 提交可触发自动发布（低风险项） |
| 💎 Elite Scout | 5000+ | 平台顾问权限 + 直接联系团队 |

### 6.3 信用衰减

- 30 天无任何贡献 → 活跃度归零
- 90 天无任何贡献 → 信用分 ×0.8
- 180 天无任何贡献 → 信用分 ×0.5
- 300 天无任何贡献 → 标记为"休眠 Scout"

## 七、反作弊机制

### 7.1 自动检测

| 检测维度 | 方法 | 阈值 |
|---------|------|------|
| 提交频率 | 同账号短时间内大量提交 | >20/小时 触发审查 |
| URL 相似度 | 提交的 URL 与前人高度相似 | Levenshtein < 3 |
| 账号关联 | 同 IP / 同设备多账号 | >2 账号 触发审查 |
| 时段异常 | 非正常作息时段大量操作 | 凌晨 2-5 点高频率 |

### 7.2 人工审核触发条件
- 自动检测系统标记为可疑
- 新账号首次提交
- 提交者的历史采纳率 < 30%
- 提交涉及高风险类型（付费、KYC）
