# 06 — 可行性评估

## 一、市场可行性

### 1.1 市场需求验证

**AI 项目发现需求确实存在：**

| 信号 | 来源 | 说明 |
|------|------|------|
| GitHub Trending 日访问量 | 公开数据 | 开发者习惯每天浏览 Trending |
| HN Show 板块活跃度 | 公开数据 | 每天数十个 Show HN AI 项目 |
| "AI tools directory" 搜索量 | Google Trends | 持续上升趋势 |
| CookieDAO 用户量 | 公开报道 | 验证了"聚合+评分"模式的可行性 |
| ProductHunt AI 分类 | 公开数据 | AI 新品的集中发现渠道 |

**结论**：AI 从业者对项目发现工具有真实需求，但市场上缺少针对 AI 领域的深度聚合平台。

### 1.2 竞争格局

| 类型 | 竞品 | TuringScout 差异化 |
|------|------|-------------------|
| 代码平台 | GitHub Trending / Explore | AI 专属筛选 + 多源数据 + 机会维度 |
| 开发者社区 | HN Show / ProductHunt | 数据驱动评分 + 持续追踪 |
| AI 目录 | "AI Tools Directory" 类网站 | 自动抓取而非人工登记 + 评分算法 |
| 模型平台 | Hugging Face | 更广覆盖面（非仅限模型）+ 社交热度 |
| Web3 情报 | CookieDAO | AI 领域定位（非代币/交易） |

### 1.3 市场风险

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| AI 赛道拥挤，新平台难以突围 | 中 | 专注细分（AI 项目机会），差异化定位 |
| 大厂可能做类似产品 | 中 | 社区和数据壁垒，先发优势 |
| 用户获取成本高 | 中 | SEO + 内容分发低成本策略 |
| 市场周期下行（AI 热度降低） | 低 | 技术趋势追踪本身是长期需求 |

## 二、技术可行性

### 2.1 数据可获取性评估

| 数据源 | 可用性 | API 限制 | 替代方案 |
|--------|--------|---------|---------|
| GitHub API | ✅ 可用 | 5000 req/h (认证) | GitHub Archive |
| GitHub Trending | ✅ 可抓取 | 非官方 API，需爬虫 | GitHub Search API |
| Hacker News API | ✅ 可用 | 无限制 | Firebase API |
| Hugging Face API | ✅ 可用 | Rate limit 较低 | 爬虫 |
| ProductHunt API | ⚠️ 受限 | GraphQL API，需认证 | 爬虫 |
| Reddit API | ✅ 可用 | 免费额度充足 | Pushshift |
| Twitter API | ⚠️ 受限 | Basic $100/月 | Nitter/爬虫 |
| 知乎 | ⚠️ 需爬虫 | 无公开 API | 爬虫 |
| ArXiv API | ✅ 可用 | 无限制 | OAI-PMH |

**结论**：核心数据源（GitHub、HN、ArXiv）数据可获取。Twitter 和知乎需要爬虫或付费 API，可推迟到 V1.5 或 V2。

### 2.2 技术难点评估

| 难点 | 难度 | 解决思路 |
|------|------|---------|
| AI 项目分类 | 中 | 关键词 + LLM 辅助分类，初期人工校正 |
| 项目去重/实体解析 | 中 | URL 标准化 + 名称相似度 + 人工确认 |
| 评分算法调优 | 中 | 初期简单加权，逐步引入 ML |
| 大规模数据抓取 | 低 | V1 数据量小（100+ 项目），无性能问题 |
| 实时更新 | 低 | Cron jobs 足够，无需实时流处理 |

### 2.3 核心难点判断

**V1 真正的难点不在技术，而在运营：**
1. 数据质量保证（分类准确、去重正确）
2. 评分算法能否真实反映项目质量
3. 用户是否愿意持续贡献（Scout 积极性）
4. 内容新鲜度的维持

## 三、代币经济可行性

### 3.1 当前判断：V1 不建议上线代币

**理由**：
1. **时机不成熟**：没有用户基础和流量验证，代币没有真实需求
2. **合规风险**：各国对代币监管趋严，过早发币可能面临法律风险
3. **运营负担**：代币经济需要专门的运营和做市，分散 V1 核心目标
4. **信任风险**：AI 数据平台发代币容易被质疑为"割韭菜"

### 3.2 代币经济的替代方案（V1）

- **积分系统**替代代币：降低合规风险，保留激励效果
- **积分公开透明**：积分规则公开，积分排行公开
- **明确空投承诺**：公开声明"积分将在未来代币 TGE 时按比例兑换"
- **积分不可交易**：V1 阶段积分不能转账，防止投机

### 3.3 代币上线前提条件

- 用户基础 5 万+ DAU
- 积分系统运行 6 个月以上且稳定
- 社区治理框架成熟
- 完成合规审查（至少在新加坡/香港有法律意见）
- 产品有明确的付费场景（API、高级功能）

## 四、风险分析

### 4.1 风险矩阵

| 风险类别 | 风险描述 | 可能性 | 影响 | 等级 | 缓解措施 |
|---------|---------|--------|------|------|---------|
| **赛道** | 竞品快速跟进 | 中 | 高 | 🔴 | 差异化定位 + 社区壁垒 |
| **内容** | 数据新鲜度不够 | 中 | 高 | 🔴 | 多源抓取 + 高频更新 |
| **信任** | 评分被质疑不公 | 中 | 高 | 🔴 | 算法公开 + 人工审核 |
| **增长** | 用户增长停滞 | 中 | 中 | 🟡 | SEO + 内容分发 |
| **技术** | 数据源 API 变更 | 高 | 低 | 🟢 | 多源冗余 + 爬虫备份 |
| **合规** | 数据抓取法律风险 | 低 | 中 | 🟡 | 仅抓取公开数据 |
| **AI依赖** | 过度依赖 LLM 做决策 | 中 | 中 | 🟡 | 人工审核保留 |

### 4.2 最大风险：内容新鲜度和信任

这两个风险是"一票否决"级别的：
- **新鲜度不足**：如果用户打开平台发现信息是 3 天前的，不会再来
- **信任崩塌**：如果评分被认为不准确或被项目方操纵，平台失去核心价值

**缓解措施**：
- 数据更新频率必须保证（核心源每小时更新）
- 设置"最后更新时间"可见标记
- 评分算法完全公开
- 建立审核机制防止刷分

## 五、30 天验证计划

### 验证目标
用最小成本验证核心假设：**AI 从业者愿意使用一个聚合平台来发现项目机会。**

### 验证步骤

**Week 1-2：数据聚合验证**
- 搭建数据抓取 pipeline（GitHub + HN）
- 手工整理 50 个 AI 项目机会
- 用 Notion/Google Sheets 发布一个静态榜单
- 在 HN/Reddit 分享，收集反馈

**成功标准**：帖子获得 50+ upvotes，20+ 有意义的评论

**Week 3-4：MVP 页面验证**
- 搭建简单的一页网站（项目列表 + 基本筛选）
- 继续手工维护数据
- 加入邮件订阅表单
- 在 Twitter 和中文社区推广

**成功标准**：200+ 邮件订阅，50+ DAU

**Week 5+：自动化 + 开发**
- 如果前两个阶段验证通过，启动正式 V1 开发
- 如果验证未通过，调整定位或放弃

## 六、结论

| 维度 | 评估 | 说明 |
|------|------|------|
| 市场可行性 | ✅ 可行 | 需求真实存在，竞争有差异化空间 |
| 技术可行性 | ✅ 可行 | 核心数据源可获取，难度在运营 |
| 代币可行性 | ⏸️ 暂缓 | V1 不建议上线代币，先用积分 |
| 核心风险 | ⚠️ 可控 | 内容新鲜度和信任是最大风险，需重点保障 |

**总体结论**：TuringScout V1 在技术和市场上可行。建议先完成 30 天低成本验证计划，确认 PMF 后再投入正式开发。
